# Sensor Simulation Policy

## 1. 목적

이 시뮬레이터는 실제 수위·유속 센서가 없는 단계에서 사용하는 **가상 센서 환경**이다.

목표는 실제 현장 수위를 정밀 예측하는 것이 아니라, 다음 구조를 UI/UX와 로직 차원에서 검증하는 것이다.

```text
강우량 + 막힘 상태
↓
가상 센서값 변화
↓
2D 배수 네트워크 시각화
↓
향후 YOLO + 센서 + XGBoost 통합
```

## 2. 실제 수리해석 모델이 아닌 이유

이 앱은 SWMM, PySWMM, CFD 기반 시뮬레이터가 아니다.

사용하지 않는 이유:

1. 현재는 관로 길이, 관경, 경사, 유역면적, 조도계수 같은 실제 수리 파라미터가 없다.
2. SWMM/PySWMM은 `.inp` 관망 모델이 필요하며, 실시간 슬라이더 데모에는 범위가 크다.
3. 사용자가 원하는 것은 우선 센서값 변화와 물 흐름을 직관적으로 보는 MVP다.
4. SWMM 결과도 물처럼 보이는 애니메이션은 별도 Canvas 레이어가 필요하다.
5. 실제 현장 데이터 없이 만든 SWMM 모델도 결국 가상 모델이므로 정확도 착시가 생길 수 있다.

SWMM/PySWMM은 향후 고도화 후보로 남긴다.

## 3. 막힘 위치 구분

### 상부 막힘

배수구 위쪽, 그레이팅 위, 도로면에 쓰레기가 쌓인 상태다.

- 도로 위 물고임 증가
- 배수구 유입량 감소
- 내부 관로 수위는 낮거나 천천히 증가할 수 있음
- 아래 관로가 멀쩡해도 물이 지하로 못 내려갈 수 있음
- 이미 관로 안에 들어온 상류 흐름은 하류로 계속 지나갈 수 있음
- 비가 그친 뒤에는 증발, 옆 방향 흘러감, 주변 배수 같은 단순 dry-down으로 도로 물고임이 서서히 감소함

### 내부 막힘

배수구 내부나 관로에 이물질이 있는 상태다.

- 물은 배수구 안으로 들어감
- 관로 내부 수위 증가
- 관로 유속 감소
- 정체 가능성 증가

### 복합 막힘

상부와 내부 문제가 동시에 존재한다.

- 도로 위 물고임 증가
- 내부 수위 증가
- 유속 저하
- 가장 직관적으로 위험해 보이는 상태

## 3.1. 단순 관로 네트워크 가정

현재 MVP는 `DRAIN_A → DRAIN_B → DRAIN_C → OUTFALL` 순서의 단순 주관로를 가정한다.

- 각 배수구의 도로 위 물고임과 유입량은 해당 지점의 상부 막힘에 의해 결정된다.
- A에서 관로로 들어간 물은 B, C 지점 아래 관로를 지나 하류로 이동할 수 있다.
- B나 C가 상부 막힘이어도, 그 막힘은 로컬 유입구를 막는 것이지 주관로 자체를 막는 것은 아니다.
- 내부 막힘은 해당 구간의 관로 통과 흐름과 유속을 낮추고, 관로 수위를 높이는 방식으로 표현한다.
- 이 네트워크 전달도 실제 수리해석이 아니라 센서 패턴을 직관적으로 보여주기 위한 단순화다.

## 4. 핵심 센서값

| 변수 | 의미 |
|---|---|
| `surface_water_level` | 도로 위 물고임 정도 |
| `inlet_flow` | 배수구 안으로 들어가는 물의 양 |
| `surface_recession` | 비가 약해지거나 그친 뒤 도로 물고임이 자연 감소하는 양 |
| `pipe_water_level` | 내부 관로 수위 |
| `pipe_flow_speed` | 내부 관로 유속 |
| `pipe_flow_rate` | 내부 관로 유량 |
| `surface_blockage` | 상부 막힘 정도 |
| `internal_blockage` | 내부 막힘 정도 |
| `stagnation_score` | 정체 가능성을 설명하기 위한 파생 점수 |

## 5. 해석 정책

중요한 정책:

```text
수위 높음 + 유속 높음 = 배수 진행 중
수위 높음 + 유속 낮음 = 정체 의심
도로 위 물고임 높음 + 유입량 낮음 = 상부 유입 막힘 의심
```

## 6. 시각화 정책

Canvas 시각화는 실제 유체역학 시뮬레이션이 아니라 **visual metaphor**다.

- 도로 위 물웅덩이 = `surface_water_level`
- 배수구로 떨어지는 물방울 = `inlet_flow`
- 파이프 안 물방울 속도 = `pipe_flow_speed`
- 물방울 밀도 = `pipe_flow_rate`
- 쓰레기 조각 = `blockage_severity`
- 경고 pulse = `stagnation_score`

계산값 자체는 그대로 유지하고, 시각적 과장은 `visual_*` 계층에서만 사용한다.

## 7. 목업 센서 Timeseries 정책

시나리오 기반 timeseries는 실제 센서 API가 아니라, 현재 가상 센서 모델을 여러 time step으로 실행한 JSON 목업이다.

- 각 snapshot은 `schema_version`, `network`, `units`, `readings`, `measurements`, `derived`, `status`를 유지한다.
- timeseries payload는 여러 snapshot과 flat records를 함께 제공한다.
- `rain_stops` 시나리오는 비가 그친 뒤 `surface_recession`으로 도로 물고임이 서서히 줄어드는지 확인하기 위한 데모다.
- `network_passthrough` 시나리오는 B/C가 상부 막힘이어도 A에서 이미 관로로 들어간 흐름이 하류로 전달되는지 확인하기 위한 데모다.
- 시나리오는 센서 패턴과 API 계약을 검증하기 위한 목업이며, CSV 학습 데이터 생성기나 실제 수리해석 엔진이 아니다.

## 8. 센서 현실감 옵션 정책

센서 현실감 옵션은 실제 센서 API 연결이 아니라, 목업 payload에 센서 품질 문제를 흉내 내기 위한 후처리다.

- 기본값은 deterministic 목업이며 measurement 값이 임의로 흔들리지 않는다.
- `noise`, `missing`, `stale`, `spike`, `stuck`, `delay`는 API request에서 선택적으로 켤 수 있다.
- `seed`를 지정하면 같은 request는 같은 목업 센서 품질 패턴을 재현한다.
- 현실감 옵션은 `measurements`에만 적용하고, `blockage`, `status`, `derived`는 깨끗한 시뮬레이션 기준값으로 유지한다.
- 각 reading은 `quality_flags`와 `measurement_quality`로 어떤 센서 품질 문제가 섞였는지 표시한다.
- 이 기능은 센서 API 계약과 UI 결측/이상값 대응을 검증하기 위한 것이며, 실제 센서 보정 모델이 아니다.
